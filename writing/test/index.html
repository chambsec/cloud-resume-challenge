# How a Cloud Resume Challenge Became My Digital Brand

I started the Cloud Resume Challenge expecting to check a box on my cloud learning checklist. What I found instead was a framework for understanding how cloud engineering actually works in production systems.

The challenge itself is straightforward on the surface. Host a resume on AWS. Add a visitor counter. Automate the deployment. It's designed by Forrest Brazeal to force hands-on contact with the services and patterns that real cloud teams use every day. But the constraint of building something real and public changed how I approached each decision.

When you build something that other people can actually visit, you can't cut corners on the architecture. A visitor counter isn't just a Lambda function logging to CloudWatch. It needs to persist data correctly. It needs to scale without breaking. It needs to stay online while you deploy new code.

I used S3 for static hosting because that's what S3 was built for. CloudFront in front of it gave me global distribution and forced me to understand certificate management with ACM. Lambda handled the counter logic, DynamoDB held the state, and API Gateway connected the frontend to the backend.

The real education wasn't in picking these services. It was in understanding why they belong together. Why you don't store state in Lambda. Why CloudFront sits in front of S3 even for a personal project. Why IAM policies should be as narrow as possible, even when nobody's attacking you yet.

Infrastructure as code transformed how I thought about the whole system. Writing Terraform forced me to be explicit about every resource, every relationship, every permission. You can't hand-click your way through infrastructure as code. You have to know what you're building before you write it.

The CI/CD pipeline came next. Pushing code to a repository triggered an automated build, ran tests, and deployed to AWS without human intervention. This isn't a feature for large teams. This is a habit I needed for any system I'd ever own, whether I'm working alone or in a company.

What made this different from a tutorial was that failure mattered. If my site went down, that was my failure. If the visitor counter broke, I had to fix it. That pressure pushed me to add monitoring and understand CloudWatch logs in a way that following a guide never would have.

A year into running this, I realized the resume itself had become secondary. The infrastructure and the systems I'd built to maintain it were the actual resume. Every commit showed my decisions. Every deployment showed I understood automation. The project became proof of what I could do.

That's when I pivoted it from a resume hosting exercise to a platform for publishing my actual work. Writing about what I've built. Documenting the systems I operate. Hosting the projects themselves on the same infrastructure that proved I could build it.

The Cloud Resume Challenge gave me more than cloud certifications or portfolio content. It gave me a real system to operate and improve. A place to learn that doesn't exist in coursework. A framework for understanding what I actually know and what I still need to learn.

If you're considering it, don't treat it as a checkbox. Treat it as the start of owning something real on the internet. That changes everything.
